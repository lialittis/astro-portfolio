### [Six mistakes in ERC-4337 smart accounts](https://blog.trailofbits.com/2026/03/11/six-mistakes-in-erc-4337-smart-accounts/) - March 11, 2026

<p>Account abstraction transforms fixed “private key can do anything” models into programmable systems that enable batching, recovery and spending limits, and flexible gas payment. But that programmability introduces risks: a single bug can be as catastrophic as leaking a private key.</p>
<p>After auditing dozens of ERC‑4337 smart accounts, we’ve identified six vulnerability patterns that frequently appear. By the end of this post, you’ll be able to spot these issues and understand how to prevent them.</p>
<h2 id="how-erc-4337-works">How ERC-4337 works</h2>
<p>Before we jump into the common vulnerabilities that we often encounter when auditing smart accounts, here’s the quick mental model of how ERC-4337 works. There are two kinds of accounts on Ethereum: externally owned accounts (EOAs) and contract accounts.</p>
<ul>
<li>
<p><strong>EOAs</strong> are simple key-authorized accounts that can’t run custom logic. For example, common flows like token interactions require two steps (approve/permit, then execute), which fragments transactions and confuses users.</p>
</li>
<li>
<p><strong>Contract accounts</strong> are smart contracts that can enforce rules, but cannot initiate transactions on their own.</p>
</li>
</ul>
<p>Before account abstraction, if you wanted wallet logic like spending limits, multi-sig, or recovery, you&rsquo;d deploy a smart contract wallet like Safe. The problem was that an EOA still had to kick off every transaction and pay gas in ETH, so in practice, you were juggling two accounts: one to sign and one to hold funds.</p>
<p>ERC-4337 removes that dependency. The smart account itself becomes the primary account. A shared <code>EntryPoint</code> contract and off-chain bundlers replace the EOA&rsquo;s role, and paymasters let you sponsor gas or pay in tokens instead of ETH.</p>
<p>Here&rsquo;s how ERC-4337 works:</p>
<ul>
<li>
<p>Step 1: The user constructs and signs a <code>UserOperation</code> off-chain. This includes the intended action (<code>callData</code>), a nonce, gas parameters, an optional paymaster address, and the user&rsquo;s signature over the entire message.</p>
</li>
<li>
<p>Step 2: The signed <code>UserOperation</code> is sent to a bundler (think of it as a specialized relayer). The bundler simulates it locally to check it won&rsquo;t fail, then batches it with other operations and submits the bundle on-chain to the <code>EntryPoint</code> via <code>handleOps</code>.</p>
</li>
<li>
<p>Step 3: The <code>EntryPoint</code> contract calls <code>validateUserOp</code> on the smart account, which verifies the signature is valid and that the account can cover the gas cost. If a paymaster is involved, the <code>EntryPoint</code> also validates that the paymaster agrees to sponsor the fees.</p>
</li>
<li>
<p>Step 4: Once validation passes, the <code>EntryPoint</code> calls back into the smart account to execute the actual operation. The following figure shows the <code>EntryPoint</code> flow diagram from <a href="https://eips.ethereum.org/EIPS/eip-4337">ERC-4337</a>:</p>
</li>
</ul>

<h2 id="2-incomplete-signature-validation-specifically-the-gas-fields">2. Incomplete signature validation (specifically the gas fields)</h2>
<p>A common and serious vulnerability arises when a smart account verifies only the intended action (for example, the <code>callData</code>) but omits the gas-related fields:</p>
<ul>
<li>
<p><code>preVerificationGas</code></p>
</li>
<li>
<p><code>verificationGasLimit</code></p>
</li>
<li>
<p><code>callGasLimit</code></p>
</li>
<li>
<p><code>maxFeePerGas</code></p>
</li>
<li>
<p><code>maxPriorityFeePerGas</code></p>
</li>
</ul>
<p>All of these values are part of the payload and must be signed and checked by the validator. Since the <code>EntryPoint</code> contract computes and settles fees using these parameters, any field that is not cryptographically bound to the signature and not sanity-checked can be altered by a bundler or a frontrunner in transit.</p>
<p>By inflating these values (for example, <code>preVerificationGas</code>, which directly reimburses calldata/overhead), an attacker can cause the account to overpay and drain ETH. <code>preVerificationGas</code> is the portion meant to compensate the bundler for work outside <code>validateUserOp</code>, primarily calldata size costs and fixed inclusion overhead.</p>
<p>We use <code>preVerificationGas</code> as the example because it’s the easiest lever to extract ETH: if it isn’t signed or strictly validated/capped, someone can simply bump that single number and get paid more, directly draining the account.</p>
<p>Robust implementations must bind the full <code>UserOperation</code>, including all gas fields, into the signature, and so enforce conservative caps and consistency checks during validation.</p>

<p>Writing state in <code>validateUserOp</code> and then using it during execution is dangerous since the <code>EntryPoint</code> contract validates all ops in a bundle before executing any of them. For example, if you cache the recovered signer in storage during validation and later use that value in <code>execute</code>, another op’s validation can overwrite it before yours runs.</p>

<h2 id="4-erc1271-replay-signature-attack">4. ERC‑1271 replay signature attack</h2>
<p>ERC‑1271 is a standard interface for contracts to validate signatures so that other contracts can ask a smart account, via <code>isValidSignature(bytes32 hash, bytes signature)</code>, whether a particular hash has been approved.</p>
<p>A recurring pitfall, highlighted by security researcher <a href="https://x.com/0xcuriousapple">curiousapple</a> (<a href="https://mirror.xyz/curiousapple.eth/pFqAdW2LiJ-6S4sg_u1z08k4vK6BCJ33LcyXpnNb8yU">read the post-mortem here</a>), is to verify that the owner signed a hash without binding the signature to the specific smart account and the chain. If the same owner controls multiple smart accounts, or if the same account exists across chains, a signature created for account A can be replayed against account B or on a different chain.</p>
<p>The remedy is to use EIP‑712 typed data so the signature is domain‑separated by both the smart account address (as <code>verifyingContract</code>) and the <code>chainId</code>.</p>
<p>At a minimum, <strong>the signed payload must include the account and chain</strong> so that a signature cannot be transplanted across accounts or networks. A robust pattern is to wrap whatever needs authorizing inside an EIP‑712 struct and recover against the domain; this automatically binds the signature to the correct account and chain.</p>

<h2 id="5-reverts-dont-save-you-in-erc4337">5. Reverts don’t save you in ERC‑4337</h2>
<p>In ERC-4337, once <code>validateUserOp</code> succeeds, the bundler gets paid regardless of whether execution later reverts. This is the same model as normal Ethereum transactions, where miners collect fees even on failed txs, so planning to “revert later” is not a safety net. The success of <code>validateUserOp</code> commits you to paying for gas.</p>
<p>This has a subtle consequence: if your validation is too permissive and accepts operations that will inevitably fail during execution, a malicious bundler can submit those operations repeatedly, each time collecting gas fees from your account without anything useful happening.</p>
<p>A related issue we’ve seen in audits involves paymasters that pay the <code>EntryPoint</code> from a shared pool during <code>validateUserOp</code>, then try to charge the individual user back in <code>postOp</code>. The problem is that <code>postOp</code> can revert (bad state, arithmetic errors, risky external calls), and a revert in <code>postOp</code> does not undo the payment that already happened during validation. An attacker can exploit this by repeatedly passing validation while forcing <code>postOp</code> failures by withdrawing his ETH from the pool during the execution of the <code>userOp</code>, for example, and draining the shared pool.</p>
<p>The robust approach is to never rely on <code>postOp</code> for core invariants. Debit fees from a per-user escrow or deposit during validation, so the money is secured before execution even begins. Treat <code>postOp</code> as best-effort bookkeeping: keep it minimal, bounded, and designed to never revert.</p>
<p>Here are some important considerations for ERC-4337:</p>
<ul>
<li>
<p>Make <code>postOp</code> minimal and non-reverting: avoid external calls and complex logic, and instead treat it as best-effort bookkeeping.</p>
</li>
<li>
<p>Test both success and revert paths. Consider that once the <code>validateUserOp</code> function returns a success, the account will pay for the gas.</p>
</li>
</ul>
<h2 id="6-old-erc4337-accounts-vs-erc7702">6. Old ERC‑4337 accounts vs ERC‑7702</h2>
<p>ERC‑7702 allows an EOA to temporarily act as a smart account by activating code for the duration of a single transaction, which effectively runs your wallet implementation in the EOA’s context. This is powerful, but it opens an initialization race. If your logic expects an <code>initialize(owner)</code> call, an attacker who spots the 7702 delegation can frontrun with their own initialization transaction and set themselves as the owner. The straightforward mitigation is to permit initialization only when the account is executing as itself in that 7702‑powered call. In practice, require <code>msg.sender == address(this)</code> during initialization.</p>

</li>
<li>
<p>Create separate smart accounts for ERC‑7702–enabled EOAs and non‑7702 accounts to isolate initialization and management flows.</p>
</li>

<h2 id="quick-security-checks-before-you-ship">Quick security checks before you ship</h2>
<p>Use this condensed list as a pre-merge gate for every smart account change. These checks block some common AA failures we see in audits and production incidents. Run them across all account variants, paymaster paths, and gas configurations before you ship.</p>
<ul>
<li>
<p>Use the <code>EntryPoint</code>’s <code>userOpHash</code> for validation.</p>
</li>
<li>
<p>Restrict <code>execute</code>/privileged functions to <code>EntryPoint</code> (and self where needed).</p>
</li>
<li>
<p>Keep <code>validateUserOp</code> stateless: don’t write to storage.</p>
</li>
<li>
<p>Force EIP‑712 for ERC‑1271 and other signed messages.</p>
</li>
<li>
<p>Make <code>postOp</code> minimal, bounded, and non‑reverting.</p>
</li>
<li>
<p>For ERC‑7702, allow init only when <code>msg.sender == address(this)</code>, once.</p>
</li>
<li>
<p>Add multiple end-to-end tests on success and revert paths.</p>
</li>
</ul>
